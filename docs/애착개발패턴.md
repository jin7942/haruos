## 1 핵심 설계 원칙

### 1.1 SOLID 원칙

#### 1.1.1 SRP (Single Responsibility Principle) - 단일 책임 원칙

- 클래스/모듈은 하나의 책임만 가진다
- "변경의 이유가 하나뿐이어야 한다"
- 예: UserService는 사용자 CRUD만, 이메일 발송은 EmailService로 분리

#### 1.1.2 OCP (Open-Closed Principle) - 개방-폐쇄 원칙

- 확장에는 열려 있고, 수정에는 닫혀 있어야 한다
- 새 기능 추가 시 기존 코드 수정 없이 확장 가능해야 함
- 예: 새 결제 방식 추가 시 PaymentProcessor 인터페이스 구현체만 추가

#### 1.1.3 ISP (Interface Segregation Principle) - 인터페이스 분리 원칙

- 클라이언트가 사용하지 않는 메서드에 의존하지 않아야 한다
- 범용 인터페이스보다 특화된 여러 인터페이스가 낫다
- 예: `Readable`, `Writable` 분리 vs 거대한 `FileHandler`

#### 1.1.4 DIP (Dependency Inversion Principle) - 의존성 역전 원칙

- 고수준 모듈이 저수준 모듈에 의존하지 않고, 둘 다 추상화에 의존
- 구현체가 아닌 인터페이스에 의존
- 예: Service가 JpaRepository 직접 사용 대신 Port 인터페이스 사용

### 1.2 버전 관리 (Semantic Versioning)

#### 1.2.1 형식: `MAJOR.MINOR.PATCH`

- **MAJOR**: 하위 호환성이 깨지는 변경 (API 변경, 구조 변경)
- **MINOR**: 하위 호환성 유지하며 기능 추가
- **PATCH**: 버그 수정, 사소한 개선

#### 1.2.2 버전 올리는 기준

| 변경 내용                                | 버전  |
| ---------------------------------------- | ----- |
| API 응답 구조 변경, 필수 파라미터 추가   | MAJOR |
| 새 엔드포인트 추가, 선택적 파라미터 추가 | MINOR |
| 버그 수정, 성능 개선, 리팩토링           | PATCH |

#### 1.2.3 실무 적용

- 개발 단계: `0.x.x` (언제든 깨질 수 있음)
- 정식 릴리스: `1.0.0`부터 시작
- 프리릴리스: `1.0.0-alpha.1`, `1.0.0-rc.1`

### 1.3 환경변수 주입 (Configuration Centralization)

#### 1.3.1 원칙

- 상수값, 설정값은 코드에 하드코딩하지 않는다
- 하나의 지점(Config 클래스, .env 파일)에서 주입받아 사용
- 환경별(dev/staging/prod) 설정 분리

#### 1.3.2 장점

- 환경별 설정 변경 시 코드 수정 불필요
- 민감 정보(API 키, DB 비밀번호) 코드에서 분리
- 설정값 한눈에 파악 가능

#### 1.3.3 구현 방식

| 언어/프레임워크 | 방식                                          |
| --------------- | --------------------------------------------- |
| Spring Boot     | `@ConfigurationProperties`, `application.yml` |
| Node.js         | `dotenv`, `config` 라이브러리                 |
| React           | `.env`, `import.meta.env`                     |

#### 1.3.4 주의점

- `.env` 파일은 `.gitignore`에 포함 권장
- `.env.example` 파일로 필요한 변수 목록 문서화
- 기본값은 개발 환경 기준으로 설정

### 1.4 DTO와 VO 활용 (Java Record 기반)

#### 1.4.1 DTO (Data Transfer Object)

- **목적**: 계층 간 데이터 전송 (Controller ↔ Service ↔ Client)
- **특징**: 요청/응답 단위, VO의 조합으로 구성
- **네이밍**: `*RequestDto`, `*ResponseDto 등`

#### 1.4.2 VO (Value Object)

- **목적**: 재사용 가능한 값 객체
- **특징**: 불변, 동등성 비교는 값으로 판단
- **네이밍**: `*Vo`

#### 1.4.3 내 활용 방식

```
DeviceRequest (DTO)
├── name: String
├── ipAddress: String
└── location: LocationVo (VO - 재사용)

DeviceResponse (DTO)
├── id: Long
├── deviceInfo: DeviceInfoVo (VO - 재사용)
├── status: StatusVo (VO - 재사용)
└── createdAt: LocalDateTime
```

#### 1.4.4 Java Record 사용 이유

- 불변성 보장 (final 필드)
- `equals()`, `hashCode()`, `toString()` 자동 생성
- 보일러플레이트 코드 제거
- 의도 명확 (이 객체는 데이터 전달용)

### 1.5 문서화 (주석/JavaDoc/Swagger/TSDoc)

#### 1.5.1 원칙

- 코드는 "무엇을"과 "어떻게"를 설명, 주석은 "왜"를 설명
- 공개 API는 반드시 문서화
- 자동 생성 가능한 문서 형식 사용

#### 1.5.2 언어별 도구

| 언어        | 도구            | 용도            |
| ----------- | --------------- | --------------- |
| Java/Kotlin | JavaDoc/KDoc    | 코드 문서화     |
| Java/Kotlin | Swagger/OpenAPI | REST API 문서화 |
| TypeScript  | TSDoc           | 코드 문서화     |
| Python      | docstring       | 코드 문서화     |

#### 1.5.3 필수 문서화 대상

- 공개 메서드/함수의 파라미터, 반환값, 예외
- 비즈니스 로직의 의도 (왜 이렇게 구현했는지)
- 복잡한 알고리즘의 동작 방식
- API 엔드포인트의 요청/응답 형식

#### 1.5.4 주석을 쓰지 않아도 되는 경우

- 메서드명으로 의도가 명확한 경우
- 단순 getter/setter
- 자명한 코드

## 2 공통 응답 형식 (ApiResponse Wrapper)

### 2.1 개념

모든 REST API 응답을 단일 형식으로 통일한다. 성공이든 실패든 클라이언트는 항상 동일한 구조로 응답을 파싱할 수 있다.
응답 객체 생성은 래퍼 함수를 사용하여 처리한다.

### 2.2 구성 요소

| 필드        | 타입       | 설명                           |
| ----------- | ---------- | ------------------------------ |
| `success`   | boolean    | 요청 성공 여부                 |
| `code`      | string     | 결과 코드 ("OK" 또는 에러코드) |
| `message`   | string     | 사람이 읽을 수 있는 메시지     |
| `data`      | T (제네릭) | 실제 응답 데이터               |
| `timestamp` | datetime   | 응답 시각                      |

### 2.3 왜 사용하는가

- **일관성**: 프론트엔드에서 모든 API를 동일한 방식으로 처리
- **에러 처리**: HTTP 상태 코드 + 비즈니스 에러 코드 조합
- **다국어**: 에러 코드 기반으로 클라이언트에서 메시지 매핑 가능
- **디버깅**: timestamp로 요청 추적 용이

### 2.4 주의점

- 페이징 응답은 `data` 안에 `content`, `page`, `totalElements` 등 포함
- 파일 다운로드 등 바이너리 응답은 예외

## 3 예외 전파 방식 (Exception Hierarchy)

### 3.1 개념

예외를 비즈니스 예외와 기술 예외로 구분하여 계층화한다. 발생 지점에서 처리하지 않고 `throws`로 상위로 전파하여, 예외 처리 전용 계층(GlobalExceptionHandler)에서 집중 처리한다.

### 3.2 계층 구조

```
CustomException (최상위, errorCode 포함)
├── BusinessException (4xx 계열) - 사용자가 해결 가능
│   ├── ResourceNotFoundException (404)
│   ├── DuplicateResourceException (409)
│   ├── ValidationException (400)
│   └── UnauthorizedException (401)
└── TechnicalException (5xx 계열) - 시스템 문제
    ├── ExternalApiException
    ├── DatabaseException
    └── InfrastructureException
```

### 3.3 설계 원칙

- **비즈니스 예외**: 사용자 입력 오류, 비즈니스 규칙 위반 → 사용자에게 명확한 메시지
- **기술 예외**: 인프라 오류, 외부 시스템 장애 → 로깅 후 일반적인 에러 메시지
- **모든 예외에 에러 코드 포함**: 프론트엔드 연동, 로그 검색용

### 3.4 장점

- Controller/Service에 try-catch 불필요 (코드 깔끔)
- 에러 응답 형식 통일 (ApiResponse와 연계)
- 로깅/모니터링/알림 일원화

## 4 공통 엔티티 상속 (BaseEntity)

### 4.1 개념

모든 엔티티가 상속받는 공통 필드를 정의한다. 생성/수정 일시는 JPA 콜백(@PrePersist, @PreUpdate) 또는 Auditing으로 자동 설정한다.

### 4.2 공통 필드

| 필드        | 타입          | 설정 방식                    |
| ----------- | ------------- | ---------------------------- |
| `createdAt` | LocalDateTime | @PrePersist (INSERT 시 자동) |
| `updatedAt` | LocalDateTime | @PreUpdate (UPDATE 시 자동)  |
| `createdBy` | String        | AuditorAware (선택)          |
| `updatedBy` | String        | AuditorAware (선택)          |

### 4.3 왜 사용하는가

- **감사(Audit)**: 언제, 누가 생성/수정했는지 추적
- **코드 중복 제거**: 모든 엔티티에 동일 필드 반복 방지
- **일관성**: 시간 필드 이름, 타입 통일

### 4.4 주의점

- **ID는 상속하지 않음**: 엔티티마다 ID 전략이 다를 수 있음 (AUTO_INCREMENT, UUID 등)
- **삭제 필드**: soft delete가 필요하면 `deletedAt`, `isDeleted` 추가
- **버전 관리**: 낙관적 락 필요 시 `@Version` 필드 추가

## 5 팩토리 메서드 패턴

### 5.1 개념

객체 생성 로직을 static 메서드로 캡슐화한다. 생성자 대신 의미 있는 이름의 팩토리 메서드를 사용하여 생성 의도를 명확히 한다.

### 5.2 네이밍 컨벤션

| 메서드      | 의미                   | 예시                          |
| ----------- | ---------------------- | ----------------------------- |
| `of()`      | 파라미터를 그대로 사용 | `LocalDate.of(2024, 1, 1)`    |
| `from()`    | 다른 타입에서 변환     | `Instant.from(zonedDateTime)` |
| `create()`  | 복잡한 생성 로직 포함  | `Order.create(cart, user)`    |
| `empty()`   | 빈/기본 객체           | `Optional.empty()`            |
| `valueOf()` | 문자열에서 변환        | `Integer.valueOf("123")`      |

### 5.3 왜 사용하는가

- **의도 명확**: `new Order(true, false, null)` vs `Order.createPending()`
- **불변 객체**: 생성자 대신 빌더 + 팩토리 조합
- **유효성 검증**: 생성 시점에 검증 로직 포함 가능
- **캐싱**: 자주 사용되는 인스턴스 재사용 가능

### 5.4 사용 위치

- **엔티티**: 상태별 생성 메서드 (`Order.createPending()`)
- **DTO**: 엔티티에서 변환 (`UserResponse.from(user)`)
- **VO**: 값 생성 (`Money.of(1000, Currency.KRW)`)

## 6 상태 관리 패턴 (State Machine)

### 6.1 개념

상태를 가진 엔티티는 상태 머신으로 관리한다. 상태 전이는 setter가 아닌 비즈니스 메서드로만 가능하도록 캡슐화한다.

### 6.2 상태 흐름 예시 (장애/알람)

```
PENDING → ACTIVE → ACKNOWLEDGED → CLEARED
            ↓           ↓
         (자동해제)  (수동해제)
```

### 6.3 설계 원칙

- **setter 금지**: `setStatus(ACTIVE)` 대신 `activate()`
- **전이 검증**: 허용되지 않은 전이 시 예외 발생
- **타임스탬프 자동 기록**: 상태 변경 시점 기록
- **변경자 기록**: 누가 상태를 변경했는지

### 6.4 구현 방식

- **단순**: 엔티티 내 비즈니스 메서드
- **복잡**: 별도 StateMachine 라이브러리 (Spring StateMachine 등)

### 6.5 적용 대상

- 주문 상태 (대기 → 결제 → 배송 → 완료)
- 장애 상태 (발생 → 확인 → 해제)
- 승인 상태 (요청 → 승인/반려)

## 7 공통코드 시스템 (CodeGroup + Code)

### 7.1 개념

하드코딩이나 Enum 대신 DB 기반 공통코드를 사용한다. 런타임에 코드를 추가/수정할 수 있어 재배포 없이 변경 가능하다.

### 7.2 발전 단계

```
1단계: Enum (Status.ACTIVE, Status.INACTIVE)
3단계: DB 공통코드 (코드 그룹 + 코드 테이블)
```

### 7.3 테이블 구조 예시

```
CodeGroup (그룹)
├── groupCode: "DEVICE_TYPE"
├── name: "장비 유형"
└── codes: [Code, Code, ...]

Code (코드)
├── code: "ROUTER"
├── name: "라우터"
├── sortOrder: 1
└── enabled: true
```

### 7.4 FK를 안 잡는 이유

- 코드 삭제/비활성화 시 참조 무결성 문제
- 코드값(String)으로 직접 저장하고 조회 시 조인
- 유연성 우선, 데이터 정합성은 애플리케이션 레벨에서 검증

### 7.5 캐싱 필수

- 공통코드는 자주 조회되지만 변경은 드뭄
- Redis 캐시 권장 (TTL 30분~1시간)
- 코드 변경 시 캐시 무효화

## 8 DTO 변환 패턴

### 8.1 개념

Entity와 DTO 간 변환 로직을 일관되게 처리한다. Entity를 직접 API에 노출하지 않고 DTO로 변환하여 반환한다.

### 8.2 방식 비교

| 방식            | 장점                  | 단점               | 사용 시점        |
| --------------- | --------------------- | ------------------ | ---------------- |
| DTO 내 `from()` | 간단, DTO에 로직 집중 | 복잡한 변환 어려움 | 단순 1:1 매핑    |
| Mapper 클래스   | 복잡한 로직 분리      | 파일 증가          | 여러 엔티티 조합 |
| MapStruct       | 자동 생성, 타입 안전  | 학습 비용          | 필드 많은 경우   |

### 8.3 N+1 방지

- 연관 엔티티는 필요한 필드만 추출
- 중첩 DTO 사용 (`DeviceResponse.deviceInfo: DeviceInfoVo`)
- Fetch Join 또는 @EntityGraph 활용

### 8.4 컨벤션

| 구분         | 특징                 | 예시                   |
| ------------ | -------------------- | ---------------------- |
| Request DTO  | 검증 어노테이션 포함 | `@NotNull`, `@Size`    |
| Response DTO | 불변, Builder 패턴   | `@Builder`, final 필드 |
| Entity 노출  | 금지                 | -                      |

## 9 Repository 패턴

### 9.1 개념

데이터 접근 로직을 추상화하여 비즈니스 로직과 분리한다. 영속성 기술(JPA, MyBatis 등)에 대한 의존성을 캡슐화한다.

### 9.2 계층별 역할

| 계층          | 역할               | 의존성                |
| ------------- | ------------------ | --------------------- |
| Service       | 비즈니스 로직      | Repository 인터페이스 |
| Repository    | 데이터 접근 추상화 | 영속성 기술           |
| JpaRepository | 실제 DB 접근       | JPA                   |

### 9.3 동적 검색 패턴

```sql
-- null 파라미터는 조건에서 제외
WHERE (:typeCode IS NULL OR type_code = :typeCode)
  AND (:keyword IS NULL OR name LIKE %:keyword%)
```

### 9.4 일반 레이어드 아키텍처에서

- Service → Repository 인터페이스 → JpaRepository
- 테스트 시 Mock 교체 용이

## 10 Facade 패턴

### 10.1 개념

여러 서비스/컴포넌트를 조합하여 복합 데이터를 제공하는 서비스 계층. 클라이언트(Controller)의 복잡도를 낮추고 트랜잭션 경계를 명확히 한다.

### 10.2 사용 시점

| 상황                  | Facade 사용 |
| --------------------- | ----------- |
| 여러 서비스 조합 필요 | O           |
| 복잡한 비즈니스 로직  | O           |
| 트랜잭션 경계 설정    | O           |
| 단순 CRUD             | X           |
| 단일 서비스 호출      | X           |

### 10.3 예시

```
DashboardFacade
```

## 12 WebSocket 패턴

### 12.1 개념

실시간 데이터는 WebSocket으로 서버에서 클라이언트로 푸시한다. 연결 관리, 재연결, 메시지 처리를 체계적으로 구현한다.

### 12.2 백엔드 구현

- **Handler**: 연결/해제/메시지 처리
- **Session 관리**: 연결된 클라이언트 목록 관리
- **Broadcast**: 모든/특정 클라이언트에 전송
- **메시지 형식**: JSON 통일

### 12.3 프론트엔드: 커스텀 훅으로 분리

#### 12.3.1 왜 훅으로 분리하는가

- **관심사 분리**: 컴포넌트는 UI만, 훅은 연결/상태 관리만
- **재사용성**: 여러 컴포넌트에서 동일한 WebSocket 로직 공유
- **테스트 용이**: 훅 로직을 독립적으로 테스트 가능
- **복잡도 감소**: 컴포넌트 코드가 깔끔해짐

#### 12.3.2 훅의 책임

| 책임        | 설명                                    |
| ----------- | --------------------------------------- |
| 연결 관리   | connect(), disconnect(), 연결 상태 추적 |
| 자동 재연결 | 끊김 감지 → 지수 백오프로 재연결 시도   |
| 세션 관리   | 인증 토큰 전달, 세션 ID 관리            |
| 메시지 처리 | 수신 메시지 파싱, 콜백 호출             |
| 생명주기    | 컴포넌트 마운트/언마운트에 따른 정리    |

#### 12.3.3 훅 설계 예시

```
useWebSocket(url, options)
├── 반환값
│   ├── connectionState: 'connecting' | 'connected' | 'disconnected' | 'error'
│   ├── lastMessage: 마지막 수신 메시지
│   ├── sendMessage: 메시지 전송 함수
│   └── reconnect: 수동 재연결 함수
├── 내부 로직
│   ├── useEffect: 연결 초기화 및 정리
│   ├── useRef: WebSocket 인스턴스 유지
│   └── useCallback: 메모이제이션된 핸들러
```

#### 12.3.4 자동 재연결 전략

```
연결 끊김 → 1초 대기 → 재연결 시도
         ↓ 실패
         2초 대기 → 재연결 시도
         ↓ 실패
         4초 대기 → 재연결 시도
         ↓ ... (지수 백오프, 최대 30초)
```

- **지수 백오프**: 실패할수록 대기 시간 증가 (서버 부하 방지)
- **최대 재시도**: 무한 반복 방지 (5~10회 후 포기)
- **성공 시 리셋**: 연결 성공하면 대기 시간 초기화

#### 12.3.5 세션/인증 관리

- **연결 시 토큰 전달**: URL 쿼리 또는 첫 메시지로 인증
- **토큰 갱신**: 토큰 만료 시 재인증 후 재연결
- **세션 유지**: heartbeat/ping-pong으로 연결 유지 확인

### 12.4 고려사항

- 연결 끊김 감지 (heartbeat/ping-pong)
- 메시지 유실 처리 (시퀀스 번호, 재전송)
- 인증 (연결 시 토큰 전달)
- 다중 연결 관리 (여러 토픽 구독 시)

## 13 캐시 전략

### 13.1 개념

자주 조회되지만 변경이 적은 데이터는 캐싱한다. 캐시 적중률과 데이터 정합성 사이의 균형을 맞춘다.

### 13.2 캐시 적용 대상 (TTL 권장값)

| 데이터      | TTL        | 이유               |
| ----------- | ---------- | ------------------ |
| 공통코드    | 30분~1시간 | 거의 변경 없음     |
| 설정 정보   | 1시간      | 변경 드묾          |
| 사용자 세션 | 30분       | 보안상 적절한 만료 |
| 집계 데이터 | 5분        | 실시간성 필요      |
| 목록 조회   | 1~5분      | 변경 빈도에 따라   |

### 13.3 캐시 무효화 전략

| 전략        | 설명                   | 사용 시점   |
| ----------- | ---------------------- | ----------- |
| TTL 만료    | 시간 경과 시 자동 만료 | 기본        |
| 명시적 삭제 | CUD 시 해당 캐시 삭제  | 정합성 중요 |
| 전체 무효화 | 관련 캐시 모두 삭제    | 구조 변경   |

### 13.4 Spring 캐시 어노테이션

| 어노테이션    | 동작                                  |
| ------------- | ------------------------------------- |
| `@Cacheable`  | 캐시 있으면 반환, 없으면 실행 후 캐시 |
| `@CacheEvict` | 캐시 삭제                             |
| `@CachePut`   | 항상 실행하고 결과 캐시               |
