# RFC-001: 무료 사용자 격리 전략 (Pool vs Silo)

## 요약

HaruOS는 현재 **사용자 본인의 AWS 계정에 테넌트 인프라를 자동 프로비저닝하는 단일 모델(Silo)**만 지원한다. 이 모델은 엔터프라이즈 고객에게는 적합하지만, **무료 체험자·SMB·개인 사용자에게는 진입 장벽이 지나치게 높고**, 동시에 **수익 모델상 허점**(체험 후 구독 해지로 인프라만 챙기고 이탈 가능)이 존재한다.

본 RFC는 **이중 트랙(Pool + Silo) 모델 도입**을 제안한다. 무료/체험 사용자는 HaruOS 본사 AWS에 논리적으로 격리된 인스턴스로 제공하고, 정식 계약 기업은 현행 모델(고객사 AWS에 물리적 격리)을 유지한다.

## 배경

### 현재 모델의 구조

- 사용자가 콘솔에서 테넌트 생성 → 본인 AWS 계정에 CloudFormation으로 Trust Role 설치 → HaruOS가 STS AssumeRole → Terraform/Ansible로 RDS·ECS·ALB·S3 프로비저닝
- 비용 구조:
  - HaruOS에게: 월 $19/테넌트 (Stripe 구독, 14일 trial)
  - AWS에게: 사용자가 직접 부담 (idle 기준 월 $100~150, AI 사용량 따라 $200~500+)

### 식별된 문제

**문제 1 — 진입 장벽이 높다**
- 가입에 AWS 계정·결제카드·Role 설정·기술 이해도가 필요
- 1인 자영업자·개인 사용자·"한 번 써볼래요" 수요를 전혀 잡지 못함
- "월 $19" 가격표는 실제로는 $120+ TCO를 숨김 → 신뢰 손상 위험

**문제 2 — 수익화 허점** ([`apps/console-api/src/modules/billing/billing.service.ts:147`](../../../apps/console-api/src/modules/billing/billing.service.ts) 검증)
- 구독 취소 시 코드 동작: Stripe 취소 + DB `status='CANCELLED'`만 기록
- TenantEntity.suspend(), Terraform destroy 등 **인프라 정리 호출 없음**
- tenant-api(테넌트 앱) 내 console-api 라이선스 체크 코드: **0건** (`grep subscription|billing|license`)
- 즉 사용자가 trial 기간 안에 인프라 셋업 → 구독 취소 → HaruOS는 단 $0도 수금하지 못한 채 사용자는 자기 AWS에 깔린 시스템을 계속 사용 가능
- BSL 1.1 라이센스(4년 후 Apache 2.0 자동 전환)라 라이선스 키 우회도 합법적으로 가능해질 수 있음

### 왜 지금 논의가 필요한가

- 현재 코드베이스는 골격 단계(어댑터 다수 Stub). **DB 스키마와 인증 흐름을 손대기 가장 좋은 시점**
- 무료 트랙 없이 마케팅하면 깔때기 위쪽이 비좁아 영업 리드 자체가 안 모임
- 한 번 단일 트랙으로 출시한 뒤 멀티 트랙으로 전환하면 DB 마이그레이션 부담이 크게 증가

## 검토 선택지

### 옵션 1 — 현행 유지 (Silo only, BYOC)

모든 사용자가 본인 AWS에 인프라를 깐다.

- **장점**:
  - 구현 변경 없음
  - 데이터 주권이 고객에게 있음 → 규제 산업(의료·금융) 영업에 강함
  - HaruOS는 인프라 비용·AI 비용을 짊어지지 않음 (적자 리스크 0)
- **단점**:
  - 무료 체험 경로 부재 → 깔때기 위쪽이 비좁음
  - 수익화 허점 그대로 (위 문제 2)
  - 1인·SMB 시장 포기

### 옵션 2 — 이중 트랙 (Pool + Silo) **(추천)**

체험·무료 사용자는 HaruOS 본사 AWS의 공유 인프라에 논리적 격리로 제공하고, 정식 계약 기업만 본인 AWS에 Silo 배포.

| 트랙 | 대상 | 인프라 위치 | 격리 방식 | 비용 부담 |
| --- | --- | --- | --- | --- |
| **Pool (체험)** | 개인, SMB, 영업 리드 | HaruOS 본사 AWS | DB의 `tenant_id` + Row-Level Security | HaruOS (한도 내 무료, 초과 시 유료 전환) |
| **Silo (엔터프라이즈)** | 정식 계약 기업 | 고객사 AWS | 별도 RDS / ECS / S3 | 고객사 |

깔때기:
```
체험 가입 (Pool) → 가치 체감 → 영업 컨택 → 엔터프라이즈 전환 (Silo)
   넓게 받음              마케팅 콘텐츠         계약·온보딩
```

- **장점**:
  - 진입 장벽 사실상 0 (이메일만 있으면 가입)
  - 검증된 패턴 (GitLab·Sentry·Mattermost·Cal.com·n8n 모두 채택)
  - 체험 사용 패턴 → 영업 리드 발굴 자동화 가능
  - 기존 Silo 트랙은 그대로 살아 있음 (엔터프라이즈 영업 영향 없음)
  - 두 트랙이 같은 tenant-api 코드를 공유 → 유지보수 효율
- **단점**:
  - 본사 AWS 비용을 HaruOS가 부담 (특히 AI 비용 폭주 리스크)
  - DB 스키마 변경 필요 (모든 tenant 테이블에 `tenant_id` 추가 + RLS)
  - 멀티테넌트 격리 보안 책임이 본사로 옴 (한 번 사고 나면 엔터프라이즈 영업 전체가 막힘)
  - 두 트랙 운영 부담 (본사 인프라 모니터링, 무료 사용자 CS)

### 옵션 3 — Pool only (모두 본사 AWS)

Silo 트랙을 폐기하고 모두 본사 AWS에 격리.

- **장점**:
  - 단일 모델, 운영 단순
  - 사용자 입장에서도 단순 (AWS 계정 불필요)
- **단점**:
  - **데이터 주권 컨셉이 깨짐** (ADR-001과 정면 배치)
  - 규제 산업·엔터프라이즈 영업 불가
  - AI 비용 전체를 HaruOS가 부담 → 매출이 늘수록 마진 압박
  - 한 사용자의 AI 폭주가 전체 인프라 비용에 직접 영향

## 추천

**옵션 2 (이중 트랙)** 을 강력히 추천한다.

이유:
1. **검증된 패턴**. GitLab(자체 호스팅 + GitLab.com), Sentry(OSS + Sentry SaaS), Mattermost, Cal.com, n8n 등 다수의 성공 사례
2. **현재 Silo 코드 자산을 폐기하지 않으면서** 신규 시장(SMB) 진입 가능
3. **수익화 허점 해결**: Pool에서는 본사 인프라라 사용량 한도 강제 가능, Silo는 엔터프라이즈 연간 계약으로 묶음
4. **ADR-001(데이터 주권 = 핵심 가치)을 유지**하면서도 추가 트랙으로 확장
5. 코드 손대기 좋은 단계라 마이그레이션 부담 최소

## 미해결 질문

ADR 승격 전 다음 항목들이 결정되어야 한다.

- [ ] **무료 티어 한도**: 메시지 N건/월? 토큰 한도? 저장 용량? 인덱싱 문서 수?
- [ ] **AI 모델 제한**: 무료는 Claude Haiku만? Sonnet은 유료에만?
- [ ] **DB 멀티테넌시 구현 방식**: PostgreSQL Row-Level Security? 애플리케이션 레벨 필터? Schema-per-tenant?
- [ ] **유료 전환 트리거**: 한도 초과 시 사용 차단? 신용카드 등록 유도? "유료 시작" 명시적 액션 필요?
- [ ] **체험 → 엔터프라이즈 마이그레이션 도구**: 백업/Export 기능을 어디까지 자동화? 데이터 이전 시 다운타임 허용 범위?
- [ ] **엔터프라이즈 가격 책정**: $19/월 단위 구독은 Pool에만 적용? Silo는 별도 연간 계약?
- [ ] **무료 사용자 인증 강도**: 이메일 OTP만? 휴대폰 인증? 악성 사용자 대응 정책?
- [ ] **본사 인프라 SLA**: 무료 트랙도 가용성 보장? 장애 시 사과·보상 정책?
- [ ] **법적 책임**: Pool 모드에서 데이터 누출 시 책임 한계 (이용약관)
- [ ] **BSL 라이센스 영향**: 4년 후 Apache 2.0 전환 시점 Pool 모드 운영 영향 검토

## 영향

채택 시 다음 변경이 필요하다.

### 코드

- **tenant-api DB 스키마** 전면 갱신
  - 모든 도메인 테이블에 `tenant_id UUID NOT NULL` 추가
  - PostgreSQL RLS 정책 적용
  - **pgvector 검색에도 `tenant_id` 필터 강제** (안 하면 다른 회사 문서가 검색됨 → 치명적)
  - 마이그레이션: 기존 Silo 테넌트는 영향 없음(별도 DB라 그대로), Pool 트랙용 신규 마이그레이션
- **사용량 한도 미들웨어** 신설
  - 메시지 수, 토큰, 저장 용량 카운터
  - 한도 초과 시 응답 차단 또는 다운그레이드
- **백업/Export 기능 완성** (현재 [`apps/console-api/src/modules/backup/`](../../../apps/console-api/src/modules/backup/)는 골격만, 실제 Export 로직 미구현)
- **console-api에 배포 모드 분기**: Pool 등록 vs Silo 프로비저닝
- **가입 흐름 분리**: "체험 시작" (이메일만) vs "기업 문의" (영업 컨택)

### 인프라

- **본사 AWS에 Pool 트랙용 tenant-api 배포 1세트** (ECS + RDS 공유)
- **본사용 운영 도구**: 모니터링, 인시던트 대응, 무료 사용자 차단 도구

### 문서·정책

- 무료/유료 기능 매트릭스 정의
- 이용약관 갱신 (Pool 모드 데이터 처리)
- 가격 페이지 분리

### 영향 받지 않는 부분

- 기존 ADR-001(데이터 주권 컨셉)은 **Silo 트랙에서 그대로 유지**
- 기존 Stripe 결제 흐름은 Pool 트랙 유료 전환에 재사용 가능
- tenant-api 코드 자체는 양쪽에서 공유

## 다음 단계

합의 시 다음 순서로 진행한다.

1. **본 RFC 합의 후 ADR로 승격** — 이중 트랙 채택 결정
2. **후속 RFC 작성** — 미해결 질문별로 (예: RFC-002 무료 티어 한도, RFC-003 DB 멀티테넌시 구현 방식 등)
3. **Phase 1 (엔터프라이즈 트랙 완성)**: Bedrock/S3/ClickUp/이메일 어댑터 실구현, AI agent → 도메인 액션 연결, 백업/Export 완성
4. **Phase 2 (Pool 모드 준비)**: DB 스키마 마이그레이션, 사용량 한도 미들웨어, 무료 티어 가격표
5. **Phase 3 (Pool 배포)**: 본사 AWS 셋업, console-api 배포 모드 분기, 가입 흐름 분리
6. **Phase 4 (전환 경로)**: 체험 → 엔터프라이즈 마이그레이션 도구, 영업 리드 트래킹

먼저 Silo 트랙으로 레퍼런스 고객 1~2곳 확보 후, 그 매출로 Pool 개발 비용을 충당하는 순서를 권장한다 (Pool 먼저 만들면 매출 없이 AI 비용만 새는 위험).
