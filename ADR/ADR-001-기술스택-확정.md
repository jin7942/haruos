## 배경

HaruOS는 관리 콘솔 + 테넌트 앱 2개 시스템으로 구성된다.
기획안(v5.2)에서 관리 콘솔은 Next.js, 테넌트 앱은 Vite+React로 분리되어 있었고,
호스팅은 전부 AWS ECS Fargate로 설계되어 있었다.
1인 개발 환경에서 기술스택 단순화와 셀프호스팅 요구사항을 반영하여 재검토했다.

## 검토 선택지

### 1. 프론트엔드 프레임워크

**선택지 A: Next.js(관리 콘솔) + Vite+React(테넌트 앱) 분리**
- 장점: SSR/SEO 지원
- 단점: 빌드 파이프라인 2벌, 컴포넌트/타입 중복, 프레임워크 2개 유지보수

**선택지 B: Vite + React 18 통일**
- 장점: 단일 스택, 공통 코드 공유 용이, 학습/유지보수 비용 절감
- 단점: SSR 미지원

### 2. 관리 콘솔 호스팅

**선택지 A: AWS ECS Fargate**
- 장점: AWS 통합, 스케일링
- 단점: 비용, 관리 복잡도

**선택지 B: 자체 서버 + Docker Compose**
- 장점: 비용 절감, 단순한 운영, 기존 서버 활용
- 단점: 스케일링 제한 (관리 콘솔 특성상 문제 없음)

## 결정

### 프론트엔드: Vite + React 18 + TypeScript + shadcn/ui 통일

관리 콘솔은 로그인 후 사용하는 관리자 대시보드로 SEO가 불필요하다.
랜딩/마케팅 페이지가 필요하면 별도 정적 사이트로 분리한다.

### 호스팅: 관리 콘솔/백엔드는 자체 서버, 테넌트 인프라는 사용자 AWS

관리 콘솔과 백엔드는 자체 서버에서 Docker Compose로 운영한다.
테넌트 앱은 기존 설계대로 사용자 AWS 계정에 프로비저닝한다.

### 확정 기술스택

| 계층 | 기술 | 호스팅 |
| --- | --- | --- |
| 관리 콘솔 FE | Vite + React 18 + TS + shadcn/ui | 자체 서버 |
| 관리 콘솔 BE | NestJS + TypeScript | 자체 서버 |
| 관리 콘솔 DB | PostgreSQL | 자체 서버 |
| 테넌트 앱 FE | Vite + React 18 + TS + shadcn/ui | 사용자 AWS |
| 테넌트 앱 BE | NestJS + TypeScript | 사용자 AWS (ECS Fargate) |
| 테넌트 앱 DB | PostgreSQL 16 + pgvector | 사용자 AWS (RDS) |
| AI | Bedrock Claude (Sonnet → Haiku fallback) | 사용자 AWS |
| 임베딩 | Bedrock Titan Embeddings | 사용자 AWS |
| 파일 스토리지 | S3 | 사용자 AWS |
| IaC | Terraform + Ansible | 자체 서버 → 사용자 AWS |
| 컨테이너 관리 | Docker Compose | 자체 서버 |
| CI/CD | GitHub Actions | GitHub |

## 근거

1. **프론트엔드 통일**: 관리 콘솔에 SEO 불필요. 프레임워크 2개 유지보수는 1인 개발에서 비효율.
2. **셀프호스팅**: 자체 서버 보유. 관리 콘솔은 트래픽이 낮아 스케일링 불필요.
3. **Docker Compose**: K8s/K3s는 과잉. 관리 콘솔 + DB + Nginx 정도면 Compose로 충분.
4. **테넌트 AWS 유지**: 데이터 격리 컨셉 유지. 사용자 AWS에 인프라를 구축하는 것이 HaruOS 핵심 가치.

## 영향

- 기획안의 관리 콘솔 Next.js 관련 내용은 Vite+React로 대체
- 기획안의 ECS Fargate(관리 콘솔) 관련 내용은 Docker Compose로 대체
- 모노레포 구조에서 프론트엔드 공통 코드(컴포넌트, 타입, API 클라이언트) 공유 설계 필요
- Terraform 모듈에서 관리 콘솔 ECS 관련 부분 제거
