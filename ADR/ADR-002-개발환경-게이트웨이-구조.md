# ADR-002: 개발 환경 단일 게이트웨이 + 2모드 접근 구조

- **상태**: 확정
- **일자**: 2026-05-31
- **결정자**: 김진범

## 배경

개발 모드에서 각 앱(console-web 5173, tenant-web 5174, console-api 3000,
tenant-api 3001)을 포트로 직접 노출했다. 문제:

- nginx 리버스 프록시를 쓴다면서 실제로는 포트 직접 접근 → 운영 구조와 불일치
- 멀티테넌트(`{tenant}.haruos.app`)를 포트 모델로 표현 불가
- 개발 머신이 내부 서비스(공용 nginx-proxy, grafana 등)를 이미 운영 중이라
  80 포트가 점유됨
- 협업 시 다른 개발자는 그 공용 proxy/내부 DNS가 없음 → 환경 재현성 문제

## 검토 선택지

1. **포트 직접 노출 유지** → 단순하나 운영과 불일치, 멀티테넌트 표현 불가. 기각.
2. **전용 개발 VM 구축 + CI/CD 연동** → 격리는 좋으나 일상 개발에 commit→배포
   왕복이 끼어 hot-reload 이점 상실. 과한 인프라. 기각.
3. **단일 nginx 게이트웨이(Host 헤더 분기) + 환경별 오버레이** → 운영과 동일한
   진입 방식, 멀티테넌트 자연 표현, 협업/메인머신 양쪽 수용. 채택.

## 결정

개발 모드를 **단일 게이트웨이 + compose 오버레이 2종**으로 구성한다.

- **베이스** `docker-compose.dev.yml`: api/web/db/gateway 정의. 게이트웨이의
  포트/네트워크는 베이스에 두지 않는다(compose 오버레이가 리스트 항목을 제거하지
  못하는 한계 때문).
- **협업 표준** `docker-compose.standalone.yml` (`make up`): 게이트웨이를
  `HTTP_PORT`(기본 80)로 노출. 접속은 `*.haruos.localhost`(브라우저가 127.0.0.1
  자동 해석 → hosts 등록 불필요). 외부 의존 없이 클론 후 바로 동작.
- **메인 개발 머신** `docker-compose.internal.yml` (`make up-internal`): 게이트웨이가
  포트를 노출하지 않고 공용 nginx-proxy의 `jin-net`에 `haruos-gateway` alias로 합류.
  접속은 `*.haruos.internal`(공용 proxy가 분기). 공용 proxy는 upstream을 변수+
  resolver로 동적 해석하여, HaruOS가 꺼져 있어도 다른 상주 서비스에 영향이 없다.

게이트웨이는 `.localhost`/`.internal` 두 도메인을 모두 받아 한 conf로 양쪽을 커버한다.

## 근거

- **운영 일치**: 운영도 nginx가 도메인 기반 라우팅 → 개발도 동일 멘탈모델
- **협업 재현성**: `*.localhost`는 RFC 6761로 어디서나 127.0.0.1. hosts/DNS 설정
  없이 클론 후 `make up`만으로 동작 → 신규 개발자 온보딩 비용 최소
- **메인 머신 통합**: 기존 공용 proxy 자산 재사용. HaruOS는 개발할 때만 띄우고,
  꺼져 있어도 grafana 등 상주 서비스는 안전(resolver 동적 해석)
- **포트 충돌 회피**: 메인 머신은 게이트웨이 포트 미노출, 협업 머신은 빈 80 사용

## 영향

- `make up`(협업) / `make up-internal`(메인 머신) 두 진입점. 일상 명령은 베이스+
  standalone을 사용.
- 메인 머신은 `make proxy-install`로 공용 proxy에 `haruos.internal.conf` 등록 필요
  (`/factory/infra/nginx/conf.d/`). `make up-internal`이 자동 호출.
- Vite 게이트웨이 경유를 위해 `host:true`, `allowedHosts`, HMR clientPort 정합 필요.
- 환경변수 주입 지점이 여전히 산재(compose / ConfigModule / process.env / vite.config).
  단일 진실공급원(루트 `.env`)으로의 통합은 후속 ADR/작업에서 다룬다.
